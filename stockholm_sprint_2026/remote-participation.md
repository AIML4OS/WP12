# Remote participation
## WP12 Stockholm Sprint 2026

This document describes the intended setup for remote participation in the WP12 Stockholm sprint, 16-18 June 2026.

The sprint is primarily organised as an on-site collaborative working event in Stockholm. However, some partners have indicated interest in participating remotely. Remote participation is welcome where it can contribute to the sprint outputs without creating a complex hybrid setup for the hands-on work.

## General approach

The intended approach is:

- remote participants join the common planning and reporting sessions via Teams
- after the initial planning, participants form smaller working groups
- working groups should preferably be either fully on-site or fully remote
- mixed hybrid groups should be avoided where possible during the hands-on prototype work
- remote participants should have clear tasks, shared documentation spaces and agreed check-in points

This setup is intended to make remote participation meaningful while allowing the on-site groups to work efficiently without needing to manage hybrid collaboration continuously during the sprint.

## Suitable sessions for remote participation

Remote participation is expected to be most useful during the following sessions.

### Tuesday 16 June

- 10:00-11:00: WP12 status update, sprint objectives and use-case selection
- 11:00-12:00: Practical setup: environments, collaboration spaces and tooling
- 13:00-14:30: Definition of focus areas, implementation approaches and evaluation approach

Remote participants should join these sessions if they want to take part in use-case selection, understand the technical setup and agree on how they can contribute during the sprint.

### Wednesday 17 June

- 12:00-13:00: Status check and lunch

Remote participants can use this session to report progress, align with other groups and raise questions.

Additional remote check-ins may be added if needed, especially for remote-only working groups.

### Thursday 18 June

- 10:30-11:30: Reporting and documentation
- 11:30-12:00: Presentation of results and reflections
- 13:00-14:00: Next steps and closing discussion

Remote participants should join these sessions if they have contributed to a use case or want to follow the sprint outcomes.

## Remote-only working groups

If several participants join remotely, they may form a remote-only working group. This group should select a use case that can be worked on independently and should document its work in the same way as the on-site groups.

A remote-only group should agree on:

- use case and scope
- data or examples to use
- tools and environment
- expected output
- evaluation or comparison approach
- documentation responsibility
- check-in times with the wider sprint

## Documentation expectations

Remote participants should use the same documentation approach as on-site participants. Each use-case group should aim to document:

- use-case purpose
- data or examples used
- technical setup
- model or tools used
- evaluation approach
- results and observations
- limitations
- possible next steps
- relevance for WP12 deliverables

The documentation should be written during the sprint, not only afterwards.

## Practical requirements for remote participants

Remote participants should make sure that they have:

- access to the Teams meeting link
- access to the relevant GitHub repository or documentation space
- access to the data, examples or environment needed for their use case
- a clear understanding of whether they are joining a remote-only group or contributing in another way
- time available for the shared planning and reporting sessions

Participants who have not previously been closely involved in WP12 should read `background.md` and `use-case-planning.md` before joining the sprint.

## Limitations of the remote setup

The sprint will not be run as a fully hybrid event throughout all working sessions. The main hands-on work will take place in smaller groups, and continuous streaming of all on-site group work is not planned.

Remote participation should therefore be planned around common sessions, remote-only work, documentation contributions and agreed check-ins rather than constant participation in all on-site discussions.
